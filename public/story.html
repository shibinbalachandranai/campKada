01
The Origin

The Idea Before the Code

It started not with a laptop, but with a backpack.

Somewhere between the snow-covered ridgelines of Kedarkantha and the thin air near Everest Base Camp, a frustration was born. Every trip meant the same ritual — hours lost across a dozen websites hunting for the right tent, a reliable rental, a guide who actually knew the mountain.

"There had to be a better way." That frustration became a conviction — build it yourself. One roof. All the gear. Real expertise. No compromise on quality.

And so CampKada was conceived. Not as a startup pitch or a business plan, but as a traveller's answer to a traveller's problem.

02
Architecture

Enter Claude Code

The development began with a conversation — literally. Instead of opening a blank IDE and staring at a cursor, the founder opened Claude Code and started talking about what CampKada needed to be.

The stack was decided fast:

Next.js App Router TypeScript Tailwind CSS MongoDB Atlas Vercel Zod Validation

The brand palette was defined as Tailwind tokens — never a raw hex value in the codebase:

forest: #2D5016  // forest green
bark:   #6B3D1E  // bark brown
sand:   #C4A882  // sandy beige
cream:  #F5F0E8  // off-white

Phase 1 was deliberately scoped: browse + enquiry, no cart, no payments. A CLAUDE.md file was written to anchor every future decision — patterns, color tokens, critical files, Phase 2 compatibility notes.

03
Backend

Building the Core

The first critical file was lib/mongodb.ts — a singleton connection using the official Next.js pattern. Every API route would depend on this one file. It had to be right before anything else could work.

Next came the Enquiry schema — capturing every message a customer ever sent. Indexed on email, createdAt, and a sparse status index for the "show all new enquiries" query.

// Single endpoint. All form types converge here.
POST /api/enquiry

type: "shop" | "rental" | "service" | "general"
details: whitelisted per type via Zod
honeypot: silent bot rejection
notification: async Slack webhook

Shared UI components kept the codebase lean — one ItemCard with a variant prop instead of three separate card components. One EnquiryFormBase powering all four form variants.

04
Brand Voice

The Content Pass

The About page was where CampKada stopped being a template and started being a brand.

Placeholder copy was replaced with the real story — real mountains, real lessons. The mission and vision were rewritten with actual words. And then, pulled from the body text and rendered large, bold, forest green:

We create journeys.
We help create memories. Highlighted in forest green — given the weight it deserved.
"From the moment you pack your bag to the stories you bring back home — Camp Kada is your silent travel companion."

The brand now had a voice. And it sounded like the mountains it was born from.

06
Business Decision

The Pricing Decision

Mid-development, a business decision changed the UI. Prices were in the data, rendering on cards and detail pages. But CampKada wasn't ready to commit to fixed pricing yet — the market was still being understood.

One conversation. One clear instruction:

"Hide the prices." No pricing discussions. No debates. Just execute.

Every price span was replaced with quiet italic text: "Enquire for pricing." Stock and availability badges stayed. The structure stayed. When pricing was ready, restoring it would take minutes.

This is what a well-architected codebase enables — business decisions that don't require surgery.

07
Production

Launch Day Fires

Vercel auto-deployed on every push. The site was live at campkada.vercel.app. Then came the real-world tests.

🔥 Fire 1: Missing Images
public/images/home/ was never committed to git. Twelve images existed locally but never made it to the repository. Vercel had nothing to serve.
✓ Fix: One git add public/images/ — all twelve committed and pushed. Next deploy, images loaded.
🔥 Fire 2: Enquiry Form Error
"Something went wrong. Please try again." — every form submission failing on production.
✓ Fix: MONGODB_URI was never added to Vercel environment variables. Set it in the dashboard + Atlas Network Access opened to all IPs. Form started working immediately.
08
Analytics

Visibility Added

With the site working, the next step was knowing how people used it. Vercel Analytics was added in three moves:

# 1. Install
npm i @vercel/analytics

# 2. Import in layout.tsx
import { Analytics } from "@vercel/analytics/next"

# 3. Drop in the JSX
<Analytics />

Three lines. Zero configuration. Every page view, every navigation — now tracked and visible in the Vercel dashboard.
